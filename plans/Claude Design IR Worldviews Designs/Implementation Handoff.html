§2Hierarchy rules

Apply across all three surfaces. These keep the family consistent without forcing every page to wear the same skeleton.

  1. Rail iff peer sections. Use the sticky left-rail layout only when a page has 3+ peer sections at the same level (Field Guide, Atlas browse). Profile has one dominant artifact (the chart) and uses a no-rail layout.
  2. One dominant element per first screen. Profile = Anchored Spread chart. Atlas = the 2-up Fingerprint grid. Field Guide = the eight-axis ruled list. Nothing else may compete for first fixation.
  3. Eyebrow → title → deck. Every page header uses the same three-line opener: uppercase eyebrow with sep dots, serif title with optional italic clause, single-sentence deck under 24 words. No exceptions.
  4. Banner before title body for non-scoring surfaces. Field Guide and Atlas browse must show the small bordered "Explainer / Browse — no scoring" banner directly under the title. Profile does not need one.
  5. Ruled rows over cards. Default to ruled rows with hairline dividers; reach for bordered cards only when content is genuinely peer (start-here cells, fingerprint cards). No drop shadows on internal blocks.
  6. CTAs at the bottom, never floating. A single primary action per page, paired with at most one secondary. Stacked on mobile.
  7. Appendices visibly demoted. Anything optional uses a dashed border, an "Optional · Non-scored" pill, and lives at the bottom — never inline with peer content.
  8. Greyscale-print legible. Any element that depends on color to encode meaning fails the rule. Encode via shape, weight, position, or label first; color is reinforcement.

§4Pure styling vs layout changes

Telling Codex / Claude Code where to land each change. Style-only changes can ship as token edits; layout changes need DOM restructuring.

Pure styling tokens · CSS

  • Type scale: serif Source Serif 4 for titles, IBM Plex Sans for eyebrows/labels, IBM Plex Mono for IDs and metadata.
  • Paper palette: --paper, --paper-2, --paper-3, --rule, --rule-strong.
  • Eyebrow / title / deck pattern, banner pattern, ruled-row dividers — all reused as components, not redrawn.
  • Card chrome: 1 px solid --rule-strong, no drop shadow on inner blocks; outer frame keeps the soft 18px / -28px shadow.
  • CTA pill, footer disclosure caret, "Optional · Non-scored" pill — token-level.
  • Greyscale-print fallback: any color-coded element ships with a paired non-color encoder (shape / weight / label).

Layout changes DOM · routes

  • Profile: replace 5-strip scoring grid with single Anchored Spread chart component; rebuild lane rows with bar / anchor / label structure.
  • Profile: AI band becomes a sibling under the chart, not a column inside it.
  • Profile mobile: rework appendix region into three <details> blocks, default closed; CTAs as a stacked <nav>.
  • Atlas: change card grid from previous direction to 2-up grid-template-columns; introduce FingerprintBars component (Oversight / Capability / Openness / Multilateral).
  • Atlas: collapse multiple per-card CTAs to one shared CTA outside the grid.
  • Field Guide: introduce sticky left rail (220 px) + content column; rail anchors to in-page section ids.
  • Field Guide: futures section moves out of body flow into appendix block at the foot of the content column.
  • All three: header chrome, eyebrow/title/deck and banner are extracted to shared components so future surfaces inherit the family without copy-paste.

§6Do not change

The list below is what holds the product's correctness together. Any of these moving in implementation is a bug, not a refinement.

Hard constraints — copy · scoring · routes

  1. Archetype names & spellings. Democratic Guardrailist, Civic Risk Steward, Strategic Competitor, Industrial Statist, Open Ecosystem Builder, Multilateral Coordinator. Do not abbreviate, retitle, or pluralize anywhere user-facing.
  2. Axis names & pole labels. Public oversight (DELEGATION ↔ OVERSIGHT), Capability pace (SLOW ↔ PUSH), Openness (CLOSED ↔ OPEN), Concentration (DIFFUSE ↔ CONCENTRATED), Legitimacy & rule-setting (PROCEDURAL ↔ SUBSTANTIVE), Competition vs coordination (COORDINATE ↔ COMPETE), Harm centering (AGGREGATE ↔ CONCENTRATED), Instruments (RIGHTS ↔ INDUSTRIAL). Locked.
  3. Scoring engine and archetype assignment. Out of scope. None of these surfaces touch /api/score, /api/archetype, or any persistence of user answers.
  4. "Explainer, not assessment." Field Guide must never produce, display, or compute a per-user score. Atlas browse must never compute one either. The banner copy is canonical: "Explainer · Reading surface only — no scoring, no second assessment."
  5. Route meaning. /profile = the user's result. /atlas = browse all archetypes. /field-guide = explainer. /compass = the assessment quiz. Do not reuse paths, redirect, or split routes during this work.
  6. Compass quiz UI. Untouched. Do not restyle the quiz to match this family in this pass — that is a separate decision.
  7. Futures content. Stays optional and non-scored everywhere it appears. No scenario weight, no "your alignment to scenario X" copy. Ever.
  8. Numeric precision. Profile must not show numeric deltas next to lanes. Tokens are "leans" and "leans heavily". Do not reintroduce numbers as tooltips.
  9. Beta marker. The Beta tag in the brand mark stays on every surface.
